# Screen 3 — Session Detail / Pipeline

**Статус:** draft v0.1
**Автор:** обсуждение с пользователем, 2026-04-11
**Область:** центральный экран приложения, через который проходят 80% действий пользователя.

---

## 1. Назначение

Показать одну сессию кампании (project) как вертикальный конвейер из четырёх
блоков: откуда берём данные → через что их пропускаем → где смотрим прогресс →
куда пишем результат. Пользователь нажимает одну кнопку `Запустить` и получает
единый транскрипт.

Экран должен читаться за 5 секунд game master'ом, который не знает слов «VAD»,
«RNNT», «compute type». Весь жаргон — за тултипами `[?]`.

---

## 2. Mental model

- **Project** — одна кампания. Папка на диске. Хранит ростер, настройки,
  **список источников по умолчанию**, историю сессий.
- **Session** — одна игровая встреча. Подпапка внутри project. Имеет свои
  входные файлы, свой кэш парсеров, свой итоговый merged.txt.
- **Source / Merger / Renderer** — три архитектурных слоя, каждый из
  которых представлен отдельным блоком на экране.
- **Module UI Contract** — каждый модуль (source/merger/renderer) несёт с
  собой две UI-проекции: компактную «карточку дома» и развёрнутую
  «runtime-панель» для блока обработки. См. `docs/architecture/module-ui-template-di.md`.

---

## 3. Структура экрана — четыре вертикальных блока

```
┌─────────────────────────────────────────────────┐
│ 1 · ИСТОЧНИКИ              [+ добавить источник]│
│   N карточек source-модулей                     │
│   по умолчанию — пусто, заполняется вручную    │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│ 2 · МЕРЖЕР                      [Настроить]     │
│   одна строка — имя мержера                     │
│   в MVP зафиксирован на timeline-v1              │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│ 3 · ОБРАБОТКА       [▶ Запустить обработку]     │
│   idle  — только кнопка                         │
│   run   — runtime-панель активного модуля       │
│   done  — сводка всех этапов                    │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│ 4 · ВЫВОД                       [Настроить]     │
│   один файл merged.txt (MVP)                    │
└─────────────────────────────────────────────────┘
```

Сверху блоков — обычный breadcrumb проекта и таб-бар сессии
(`Обработка` / `Транскрипт` / `Журнал` / `Настройки сессии`). Текущий экран —
таб `Обработка`.

---

## 4. Блок 1 · Источники

### Поведение

- **Пустой на старте.** При создании новой сессии блок 1 содержит только
  кнопку `[+ добавить источник]` и подсказку «Добавьте первый источник —
  например, папку с записями Craig».
- **Дефолты живут в проекте, не в сессии.** Project хранит список
  «обычных источников для этой кампании». При создании новой сессии блок 1
  инициализируется из project defaults. Пользователь может добавлять,
  удалять, менять карточки — это не влияет на project defaults, только на
  текущую сессию. Управление project defaults — в `Настройки проекта`.
- **Кнопка `[+ добавить источник]`** открывает модальное меню «какой тип
  источника добавить»: `Аудио` / `Foundry VTT чат` / ... (список тянется из
  реестра source-модулей).

### Карточка источника — состояния

**Idle (минимальная информация)**

- Иконка типа (`🎙` аудио, `💬` чат)
- Имя источника («Аудио», «Foundry VTT чат»)
- Компактный список входных файлов (имена с discord-суффиксами, как их
  положил Craig) с мелкими иконками
- Кнопка `[Настроить]` справа внизу
- Статус-чип «готов» / «нужны файлы» / «ошибка»

**Running (подсвечена ярко)**

- Тот же контент, но карточка выделена рамкой accent-цвета
- Статус-чип меняется на «● в работе»
- Карточка **не раскрывается** — раскрытие живёт в блоке 3

**Dimmed (другой модуль работает)**

- Контент тот же, цвет приглушён (~50% прозрачности)
- Карточка неинтерактивна (клики на `[Настроить]` проигнорированы или
  показывают тултип «недоступно во время обработки»)

### UI-темплейты

Каждая карточка источника рендерится не своим собственным виджетом, а
**общим темплейтом**, выбранным через DI в `ui_config` модуля:

- `audio_source_template` — для ЛЮБОГО audio-backend (GigaAM, Whisper,
  sherpa-onnx, faster-whisper, ...)
- `chat_source_template` — для ЛЮБОГО парсера чата (FVTT, Discord export,
  Roll20, ...)

См. `docs/architecture/module-ui-template-di.md`.

## 5. Блок 2 · Мержер

### Поведение

- **Одна строка,** без развернутых настроек. Только:
  - Иконка `🔀`
  - Имя мержера («timeline-v1 — объединение по временным меткам»)
  - Кнопка `[?]` с тултипом, объясняющим что делает мержер
  - Кнопка `[Настроить]` (параметры мержера — окно приоритета сдвигов,
    политика разрешения коллизий, etc.)
- **В MVP зафиксирован** на `timeline-v1`. Без выпадающего списка. Кнопка
  «Сменить» появится когда появится второй мержер — не раньше.
- **Running** — dimmed или подсвечен ярко, так же как карточки
  источников, по тем же правилам.

---

## 6. Блок 3 · Обработка (двигатель)

Центральный блок экрана. Самый важный. Может находиться в одном из четырёх
состояний:

### 6.1. Idle

- Большая кнопка `[▶ Запустить обработку]` по центру
- Подсказка мелким текстом: «Прогресс каждого источника появится здесь»
- Всё.

### 6.2. Running — один модуль за раз

Принципиально: в блоке 3 **одновременно виден ровно ОДИН модуль** — тот,
который сейчас работает. Не стек из трёх виджетов.

Последовательность в MVP:

1. Аудио-источник 1 → работает в блоке 3, карточка блока 1 ярко,
   остальные dimmed
2. Аудио-источник 2 (если есть) → работает в блоке 3
3. Чат-источник → работает в блоке 3
4. Мержер → работает в блоке 3, карточка блока 2 ярко
5. Рендерер → работает в блоке 3, карточка блока 4 ярко
6. Готово → блок 3 переходит в Done-сводку

### 6.3. Runtime-панель модуля

Это и есть **runtime widget** из Module UI Contract. Его поставляет
темплейт модуля. Для `audio_source_template` панель выглядит так:

- Имя и тип модуля вверху: «🎙 Аудио · GigaAM-v3»
- Список треков, по одному ряду на трек:
  - Имя игрока (слева), роль-бейдж (GM / Игрок / Слушатель)
  - Прогресс-бар со статусом: `█████░░░░  67%`
  - Текущая стадия (`VAD`, `ASR`, `postproc`), ETA
  - Статус: `в очереди` / `идёт` / `✓ из кэша` / `✓ готово` / `исключён`
- Текстовый резюме внизу: «Текущий этап: VAD дорожки Andrey. Кэш: 1 из 5».

Для `chat_source_template` — другая панель: «Распарсено N сообщений из M»,
превью первых реплик, etc.

Для `merger_template` — «События получены: N / M. Слияние по меткам
времени».

Для `renderer_template` — «Пишется merged.txt, 23 KB из ~80 KB».

### 6.4. Done — сводка

Когда все модули отработали, блок 3 перестаёт быть runtime-панелью
последнего модуля и превращается в **сводку всего прогона**:

- Заголовок: «✓ Готово за 14 минут 23 секунды»
- Кнопка `[▶ Перезапустить]` справа сверху
- Список этапов, по строке на каждый:
  - `• Аудио · GigaAM-v3: 5 дорожек, 3ч 47м, 12 340 событий`
  - `• Чат FVTT: 1423 события`
  - `• Мержер timeline-v1: 13 763 события в итоговом таймлайне`
  - `• Рендерер: merged.txt (84 KB)`

Сводка — единственное место в UI, где пользователь видит «отчёт о
прогоне». Логи подробнее — во вкладке `Журнал`.

### 6.5. Контролы кэша

Под runtime-панелью (или под сводкой):

- `☑ использовать кэши` [?]
- `[очистить кэш сессии]` — ghost button с подтверждением

Кэши — по-модульные, per-track для аудио. Очистка удаляет весь кэш текущей
сессии, но не трогает другие сессии проекта.

---

## 7. Блок 4 · Вывод

### Поведение

- **Idle:** текущие настройки рендерера («Формат: merged.txt») + кнопка
  `[Настроить]`. Пустое место под файл.
- **Running:** dimmed.
- **Done:** имя итогового файла + размер + количество слов, кнопки
  `[Открыть]` и `[Показать в папке]`.

В MVP поддерживается один рендерер — `txt` (merged.txt). Кнопка
`[Настроить]` открывает параметры рендерера (добавлять ли секции, как
именовать персонажей, и т.д.).

Дополнительные рендереры (`html`, `gpt-chunks`) добавляются как отдельные
source-модули — то есть пользователь может добавить `[+ добавить вывод]`
в блок 4. **Отложено на post-MVP.**

---

## 8. Явно выкинутое из MVP

- ❌ **Пауза и отмена** — кнопок нет. Запустил — ждёшь. Закрытие окна =
  graceful abort без подтверждения, но и без кнопки.
- ❌ **Параллельные прогоны модулей** — всё строго sequential. Причина:
  CPU-инференс GigaAM не даёт выигрыша от параллелизма, а синхронизация
  UI усложняется.
- ❌ **Real-time transcript preview** — транскрипт виден только после
  завершения, во вкладке `Транскрипт`.
- ❌ **Встроенный редактор** — внешний, открывается через
  `[Открыть в блокноте]`.
- ❌ **Сменный мержер** — один на MVP.
- ❌ **Множественные рендереры** — один на MVP.

## 9. `[Настроить]` — поведение и хост

Кнопка `[Настроить]` присутствует на каждой карточке источника, на
мержере, на рендерере. Клик открывает **боковой drawer справа** —
всплывающую панель, перекрывающую правые 80% окна. Основной контент
**не сдвигается** — drawer ложится поверх, как Android navigation drawer
или гамбургер-меню, а не как VS Code sidebar (который раздвигает
редактор).

### Поведение drawer

- **Ширина:** 80% от текущей ширины окна, адаптивно пересчитывается
  на `resizeEvent`
- **Scrim:** полупрозрачный слой слева (оставшиеся 20% ширины,
  `rgba(0,0,0,0.25)`), клик по scrim → закрытие drawer'а
- **Анимация:** выезжает справа за 220 мс, `OutCubic`
- **Закрытие:** scrim, Esc, крестик в хедере, `[Отмена]`, `[Сохранить]`.
  При `has_unsaved_changes() == True` — подтверждение «Сохранить изменения?»
- **Ровно один drawer** открыт в любой момент времени
- **Во время обработки** кнопки `[Настроить]` задизейблены — drawer не
  открыть

### Структура drawer

Сверху вниз:

1. **Sticky header** (~72 px) — иконка модуля + заголовок
   («Настройки · Аудио») + подзаголовок (backend + язык) + крестик
   справа. Рисует хост.
2. **Scroll area** — содержит `make_settings_panel(...)` темплейта
   модуля. Это и есть вся форма настроек.
3. **Sticky footer** (~64 px) — слева индикатор dirty, справа
   `[Отмена]` ghost + `[Сохранить]` primary. Рисует хост.

Поскольку drawer занимает 80% окна, он ощущается как «второй экран»,
не как всплывашка. Типографика и отступы — такие же как на главном
экране, без экономии.

### В Qt-терминах

- Drawer: `QFrame(parent=main_window)`, **не в layout** → работает как
  absolute-позиционированный оверлей
- Scrim: отдельный полупрозрачный `QWidget`, тоже `parent=main_window`,
  размер 20% слева, `mousePressEvent` → закрытие
- Анимация: `QPropertyAnimation(drawer, b"geometry")`, 220 мс
- Контент модуля: `QScrollArea` внутри drawer'а, `setWidget()` принимает
  то, что вернул `make_settings_panel(...)`
- Пересчёт ширины: главный `resizeEvent` → `drawer.setGeometry(...)`

Контракт модуля `make_settings_panel(parent, module, state, params)`
агностичен к тому, куда его встроили. Если в будущем drawer заменится
на что-то другое (full-screen overlay, отдельное окно, веб-интерфейс) —
хост меняется, темплейты модулей не трогаются.

См. `docs/architecture/module-ui-template-di.md` → раздел
«Хост: SettingsDrawer» и `SettingsPanelProtocol`.

## 10. Состояния ошибок

- **Источник без файлов:** на карточке красный чип «нужны файлы», кнопка
  `[▶ Запустить]` в блоке 3 **задизейблена** пока есть хоть один
  источник без входа.
- **Ошибка во время обработки:** runtime-панель этого модуля
  подсвечивается красным, следующие модули **не запускаются**, блок 3
  переходит в sub-state «Ошибка». Виден exception, ссылка «подробнее →
  Журнал».
- **Частичный результат после ошибки:** кэши всех уже завершённых
  модулей сохраняются. Юзер исправляет причину → `[▶ Запустить]` →
  пайплайн продолжается с кэшей, проблемный модуль перезапускается.

## 11. Нерешённые вопросы

1. **Project default sources** — UI для управления ими. Прописан в общем,
   но отдельного экрана ещё нет.
2. **Multi-session parallel** — можно ли запустить обработку сессии A, пока
   ты настраиваешь сессию B? MVP-ответ: нет, одна обработка на весь
   процесс.
3. **Telemetry / crash reports** — не в этом экране, но влияет на «Журнал».
   Отложено.
4. **A11y** — keyboard-only навигация пока не проработана.
