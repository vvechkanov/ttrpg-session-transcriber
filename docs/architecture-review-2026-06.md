## 1. Резюме (executive summary)

Проект в **значительно лучшем состоянии, чем заявляет его собственная
документация**. Шестислойная архитектура из `ARCHITECTURE.md` не «планируется» —
она внедрена, dependency rules соблюдаются на ~95%, есть ~440 тестов с
трёхуровневой e2e-стратегией, CI-матрица 2 ОС × 2 Python, чистая git-гигиена и
ADR-культура. Это уровень зрелости, который редко встречается в соло-проектах.

Главные проблемы — **не в коде, а вокруг него**:

1. 🔴 **Документация активно дезинформирует.** Шесть документов (README, оба
   языковых варианта, CONTRIBUTING, 00_README, 01_Как_пользоваться) дают
   нерабочие инструкции запуска через удалённый `scripts/wisper_launcher.py`.
   `TASKS.md` описывает уже выполненный рефакторинг как «текущий фокус».
   Новый контрибьютор гарантированно сломается на первом шаге.
2. 🔴 **Нет `pyproject.toml`** — версия проекта живёт в `launcher/version.py`,
   зависимости размазаны по `requirements-qt.txt`, CI-yaml и PS-скрипту,
   обещанный в CONTRIBUTING `pip install -e .[dev]` не работает.
3. 🟡 **Точечный архитектурный долг в коде**: одно нарушение dependency rule
   (`merger_worker.py`), мёртвый стаб `core/cache.py`, два реестра бэкендов,
   которые надо синхронизировать вручную, и два god-object в UI-слое
   (`TrackListModel` 659 строк, `PipelineController` 661 строка).

Рекомендуемая стратегия: **одна короткая фаза гигиены (≈2 дня), одна фаза
фундамента (≈1 неделя), дальше — продуктовые фичи** из `FEATURE_REQUESTS.md`
с параллельным точечным рефакторингом UI. Большой рефакторинг не нужен.

Десктопное приложение (Windows-first) для обработки записей TTRPG-сессий:

## 3. Архитектура: целевая vs фактическая

### 3.1 Слои и dependency rules

Целевая схема (`ARCHITECTURE.md` §3): `ui → core → {sources, mergers,
renderers} → domain`, domain pure.

**Фактическое соответствие — высокое.** Проверено grep-ом по всем импортам:

| Правило | Статус |
|---|---|
| `domain` не импортирует ничего из проекта | ✅ |
| `sources` → только `domain` | ✅ |
| `mergers` → только `domain` | ✅ |
| `renderers` → только `domain` | ✅ |
| `core` не знает про UI-тулкиты | ✅ (но см. F-C6: `timeline_window.py`) |
| `ui` → только `core` | ❌ **1 файл**: `ui/engines/merger_worker.py:32-35` импортирует `domain`, `mergers`, `renderers` напрямую |

### 3.2 Чем реальность переросла документ

`ARCHITECTURE.md` писался перед P2 и не обновлялся после QML-миграции. Чего в
нём нет, а в коде есть:

- **UI-подслои**: `ui/models/` (Qt-модели: session 1190 строк, preferences,
  registry), `ui/engines/` (QThread-workers: asr, merger, peaks, install,
  pipeline_controller), `ui/qml/` (43 компонента, ~7000 строк QML/JS).
- **Новые core-модули**: `asr.py` (фасад make_source), `file_matchers.py`,
  `timeline_window.py`, `peaks.py`, `chunking.py`, `backend_installers.py`,
  `fvtt_helpers.py` — 8 из 13 модулей core не описаны.
- **Launcher как отдельный артефакт**: два .exe (tkinter-bootstrap ≈12 МБ +
  QML-runtime ≈120 МБ), ML-стек ставится в `%APPDATA%` лениво. Нигде не
  описано как архитектурное решение (а это решение!).
- **Стадия `chunk`** в пайплайне (7 стадий вместо 6).
- ADR-017 «PySide6» не зафиксировал финальный выбор **QML** (vs Widgets).

Это не «архитектура разъехалась» — это **документ отстал**. Слои и контракты
(Source/Merger/Renderer ABC, Timeline, ScriptEvent union) соблюдены точно.

## 4. Находки

Нумерация `F-<категория><номер>` — на неё ссылается план работ в `TASKS.md`.

### Категория A — Документация (severity: 🔴 высшая)

- **F-A1. Мёртвые инструкции запуска.** `README.md:70,168`, `README.ru.md:70,168`,
  `CONTRIBUTING.md:30`, `00_README.md:20`, `01_Как_пользоваться.md:70-73`,
  `scripts/00_README.md:9` ссылаются на удалённый `scripts/wisper_launcher.py`.
  Реальный запуск: `python -m ui` (GUI) / `python -m ui.cli` (CLI).
- **F-A2. `TASKS.md` описывал выполненный план как текущий** (исправлено этим
  ревью — файл переписан).
- **F-A3. `ARCHITECTURE.md` отстал от QML-реальности** (см. §3.2). Нужны:
  секция про UI-подслои и QML, секция «Launcher vs Runtime» (два exe, два
  UI-стека, ленивый ML-стек в `%APPDATA%`), стадия `chunk`, ADR про выбор QML,
  ADR про судьбу cache (см. F-C2).
- **F-A4. `02_Статус_и_заметки.md`** — эпоха tkinter/WhisperX (исправлено этим
  ревью).
- **F-A5. `CHANGELOG.md`** не пополнялся с момента «Unreleased: hygiene» —
  пропущены все фичи #2–#7, QML-миграция, GigaAM.

### Категория B — Packaging и CI (severity: 🔴/🟡)

- **F-B1. Нет `pyproject.toml`** 🔴. Следствия: версия только в
  `launcher/version.py`; `pip install -e .[dev]` из CONTRIBUTING не работает;
  CI хардкодит `pip install PySide6 pytest pytest-qt pytest-cov numpy`;
  нет места для конфигурации ruff/mypy.
- **F-B2. CI: маркер `gui` не исключён явно** 🟡. `ci.yml:85` фильтрует только
  `not slow and not requires_asr`; `test_integration_full_pipeline.py` с
  `@pytest.mark.gui` попадает в прогон «случайно успешно» (offscreen). Надо
  сделать намерение явным: либо включить gui-маркер в фильтр, либо
  задокументировать, что offscreen-прогон желателен.
- **F-B3. CI: нет pip-кэша** 🟡 — каждый прогон тянет PySide6 (~50 МБ) заново.
  `actions/setup-python@v5` с `cache: pip` — одна строка.
- **F-B4. `launcher/build.spec` фантомные hiddenimports** 🟡 — указаны
  `core.backend_installers`, `sources.speech.*`, которые `bootstrap.py` не
  импортирует. Либо мёртвый груз (убрать), либо недокументированная фича.
- **F-B5. Linters отсутствуют** 🟡 — ruff запланирован ещё в старом плане,
  так и не появился. Dependency rules держатся только дисциплиной, а их уже
  один раз нарушили (F-C1) — ruff `flake8-tidy-imports`/`import-linter`
  автоматизирует контроль.

### Категория C — Архитектурный долг в коде (severity: 🟡)

- **F-C1. Нарушение dependency rule.** `ui/engines/merger_worker.py:32-35`
  импортирует `domain.timeline`, `mergers.script_merger`,
  `renderers.plain_text`. Лечение: фасад `core/merge_render.py` c функцией
  `merge_and_render(timeline, merger="script", renderer="plain-text") ->
  bytes` (плюс прогресс-колбэк), worker импортирует только core.
- **F-C2. `core/cache.py` — мёртвый стаб.** `_load/_save` бросают
  `NotImplementedError` (`core/cache.py:38,42`), в pipeline не подключён.
  При этом ADR-7 описывает кэш как ключевую оптимизацию (ASR = 10–30 мин).
  **Решить судьбу**: (а) реализовать инвалидацию (hash конфига + mtimes) и
  подключить в `pipeline.run` / `PipelineController`, или (б) удалить стаб и
  записать ADR «кэшем являются canonical JSON в `transcripts/`». Не оставлять
  как есть — стаб с NotImplementedError в core вводит в заблуждение.
- **F-C3. Двойной реестр бэкендов + string-dispatch.**
  `core/asr.py:98-131` — if/elif по `model_id`;
  `core/backend_installers.py` — свой hardcoded-реестр из 2 позиций;
  `sources/__init__.py` — SPEECH_SOURCES из 3. Добавление бэкенда = правки в
  3 местах + `_speech_kwargs` в `pipeline.py:180-211`. Лечение: единый
  дескриптор бэкенда (id, Source-класс, поддерживаемые опции, installer) в
  одном модуле, остальное — производные.
- **F-C4. God-objects в UI.**
  - `ui/models/session.py:375-1034` `TrackListModel` (659 строк): discovery c
    диска + UI-state + математика сегментов → проценты + запись speaker_map.
    Резать на: загрузчик (pure-функция поверх `core.file_matchers`),
    state-модель, payload-builder.
  - `ui/engines/pipeline_controller.py` (661 строка): оркестрация ASR-очереди
    + speaker_map I/O (210-353) + done-summary/chunking (544-607). Выделить
    `SpeakerMapManager` и `DoneSummaryBuilder`.
- **F-C5. Копипаста install-флоу** между `_fw_download/_fw_paths` и
  `_gigaam_download/_gigaam_paths` (~30%). Извлечь общий `InstallFlow`-каркас
  поверх уже общего `_bundle_download.py`. Делать **при добавлении третьего
  бэкенда**, не раньше (rule of three).
- **F-C6. `core/timeline_window.py` (367 строк)** — математика
  `pct_for`/window для QML-таймлайна. Это presentation-логика в core. Сейчас
  безвредно (Qt не импортирует), но при росте — перенести в `ui/` или
  выделить слой `presentation`. Низкий приоритет.
- **F-C7. `core/fvtt_helpers.py`** — lazy-import из `sources` внутри функции,
  чтобы формально не нарушать правило. Работает, но это запах: правило
  `core → sources` и так разрешено, lazy-import не нужен — выпрямить.
- **F-C8. `AppPreferences`** — 14 Q_PROPERTY × ~10 строк бойлерплейта
  (`ui/models/app_preferences.py`, 413 строк). Фабрика свойств
  (descriptor/`functools.partial`-генератор) сократит вдвое и уберёт класс
  ошибок «забыл `.sync()`/`.emit()`».
- **F-C9. QML-гигиена**: ~106 захардкоженных строк без `qsTr()` (блокер
  будущей локализации), точечные цвета мимо `Theme` (например
  `SessionTopBar.qml:25`). Делать только если англоязычная аудитория
  реально в планах — иначе зафиксировать «продукт ru-only» в README и закрыть.

## 7. Процессные рекомендации

1. **Один источник правды по плану.** `FEATURE_REQUESTS.md` — продуктовый
   бэклог (работает отлично, сохранить формат). `TASKS.md` — инженерный план
   (переписан этим ревью). Всё остальное (`00_*`, `01_*`, `02_*`) — либо
   актуализировать при каждом релизе, либо свернуть в README.
2. **Definition of Done расширить**: фича не закрыта, пока README/CHANGELOG/
   ARCHITECTURE не обновлены. Именно на этом проект и споткнулся.
3. **ADR-культура уже есть** — продолжать: каждое решение из Фазы 1 (cache,
   QML, единый реестр) фиксировать как ADR-018/019/020.
4. **CI как страж границ**: после F-B5 нарушение dependency rules должно
   валить сборку, а не ждать ревьюера.
